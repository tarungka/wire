### 1Ô∏è‚É£ Overview & Purpose

*   **High-Level Summary:** The `main` package, located in the `cmd/` directory, serves as the entry point and orchestrator for the `wire` application, a distributed stream processing platform. It handles application initialization, configuration loading, startup of core services (including Raft consensus, internode communication, HTTP API, and data store), signal handling for graceful shutdown, and manages the overall lifecycle of a node.
*   **Key Features:**
    *   Application startup and shutdown.
    *   Command-line argument parsing and configuration management.
    *   Initialization of network listeners for Raft and cluster communication.
    *   Setup of the distributed data store.
    *   Launch of the HTTP API service.
    *   Management of node participation in the cluster (bootstrapping, joining).
    *   Graceful handling of OS signals (SIGINT, SIGTERM) for clean shutdown.
    *   Logging setup and management.

### 2Ô∏è‚É£ Core Concepts & Architecture

*   **High-level architecture:** The `main` function orchestrates the startup of several key components. It establishes network listeners, initializes a data store (likely leveraging Raft for consensus via `internal/new/store`), exposes an HTTP API for external interaction (`internal/http`), and manages cluster membership and communication (`internal/cluster` and `internal/tcp` for custom muxing).
*   **Main packages and their roles:**
    *   `cmd/` (current package): Entry point, configuration, orchestration.
    *   `internal/logger`: Provides logging capabilities.
    *   `internal/tcp`: Manages low-level TCP communication, including a custom multiplexer (`tcp.Mux`).
    *   `internal/cluster`: Handles Raft consensus logic, node discovery, and cluster membership.
    *   `internal/new/store`: Implements the distributed data store, built upon the Raft layer.
    *   `internal/http`: Provides the HTTP API service.
*   **Flow summary:**
    1.  OS signals are configured for graceful shutdown.
    2.  Command-line flags are parsed, and configuration is loaded and validated.
    3.  Logging is initialized.
    4.  A TCP listener for internode communication (Mux) is started.
    5.  The Raft transport layer is set up over the Mux.
    6.  The data store (`NodeStore`) is created.
    7.  The cluster service is initialized, using its own listener from the Mux.
    8.  The HTTP service is started.
    9.  The data store is opened (which might involve Raft initialization).
    10. Cluster formation logic (bootstrapping or joining) is executed.
    11. The application waits for a shutdown signal.
    12. Upon receiving a signal, services are closed in reverse order of initialization.
*   **Architectural Pattern:** Appears to be a modular monolithic service. While it has distinct components (store, HTTP, cluster), they are tightly coupled and run within the same process. The use of Raft suggests a distributed systems architecture for data consistency.
*   **Main Components:**
    *   `Config` (struct in `cmd/init.go`): Holds all application configuration.
    *   `tcp.Mux`: Custom TCP multiplexer for handling different communication protocols (Raft, cluster messages) over a single port.
    *   `store.NodeStore` (from `internal/new/store`): Represents the node's data store and Raft participant.
    *   `httpd.Service` (from `internal/http`): The HTTP API service.
    *   `cluster.Service` (from `internal/cluster`): Manages cluster state and inter-node communication for cluster operations.
    *   `cluster.Client` (from `internal/cluster`): Used for interacting with the cluster service.

### 4Ô∏è‚É£ Data Flow & Execution

*   **Program Startup:**
    1.  Execution begins in `cmd/main.go`'s `main()` function.
    2.  `HandleSignals(syscall.SIGINT, syscall.SIGTERM, os.Interrupt)` is called to set up listeners for termination signals.
    3.  `CreateContext(sigCh)` creates a main context that will be canceled if any of the specified signals are received.
    4.  A log file (`server.log`) is opened.
    5.  `initFlags()` (from `cmd/init.go`) is called. This function defines all command-line flags, parses `os.Args[1:]`, populates a `Config` struct, and performs extensive validation. If validation fails or `--version` is requested, the program exits.
    6.  The application logo is printed.
    7.  Logging is configured using `internal/logger` based on `cfg.DebugMode` and the opened log file.
    8.  If in debug mode, PID, PPID, Hostname, and Host IP are logged.
*   **Service Initialization (Typical Flow from `main()`):**
    1.  `net.Listen("tcp", cfg.RaftAddr)`: A TCP listener is created for the internode Mux.
    2.  `startNodeMux(cfg, muxListener)`: The `tcp.Mux` is started on this listener. This Mux will handle different types of connections (Raft, cluster messages) on the same port.
    3.  `mux.Listen(cluster.MuxRaftHeader)`: A specific listener for Raft traffic is obtained from the Mux.
    4.  `cluster.CreateRaftDialer(...)` and `tcp.NewLayer(...)`: A Raft transport layer (`raftTn`) is created.
    5.  `createStore(cfg, raftTn)`: The `store.NodeStore` (which embeds Raft logic) is instantiated.
    6.  `mux.Listen(cluster.MuxClusterHeader)`: A listener for cluster service traffic is obtained from the Mux.
    7.  `clusterService(cfg, clstrLn, str, str)`: The `cluster.Service` is created and started.
    8.  `createClusterClient(cfg, clstrServ)`: A `cluster.Client` is created for this node to interact with the cluster.
    9.  `startHTTPService(cfg, str, mainCtx, clstrClient)`: The `httpd.Service` is created and started, listening on `cfg.HTTPAddr`.
    10. `str.Open()`: The data store is opened. This typically involves initializing or joining the Raft cluster.
    11. `createCluster(mainCtx, cfg, ...)`: This function handles the logic for:
        *   Bootstrapping a new single-node cluster if no join addresses or discovery mode is specified and it's a new node.
        *   Joining an existing cluster if `cfg.JoinAddresses` are provided.
        *   Using discovery services (though parts are marked `TODO`) if `cfg.DiscoMode` is set.
*   **Running State:** The application then blocks on `<-mainCtx.Done()`, waiting for the main context to be canceled (e.g., by an OS signal).
*   **Shutdown:**
    1.  When a signal is received, `mainCtx` is canceled.
    2.  `httpServ.Close()` and `clstrServ.Close()` are called to stop network services.
    3.  Optional: If `cfg.RaftClusterRemoveOnShutdown` is true, the node attempts to remove itself from the cluster (currently has a `TODO` for credentials).
    4.  Optional: If `cfg.RaftStepdownOnShutdown` is true and the node is a leader, it steps down.
    5.  `muxListener.Close()`: The main Mux listener is closed.
    6.  `str.Close(true)`: The data store is closed (which also shuts down Raft).
    7.  The log file is closed (via defer).

### 7Ô∏è‚É£ Design Rationale & Trade-offs

*   **Signal Handling First:** `HandleSignals` and `CreateContext` are called at the very beginning of `main`. This is crucial for ensuring that the application can always attempt a graceful shutdown, even if later initialization steps fail.
*   **Configuration via Flags (`pflag`):** `pflag` is used for command-line argument parsing, offering more features than the standard `flag` package (e.g., POSIX-compliant flags). The `Config` struct centralizes all settings, and the `Validate()` method ensures basic consistency before services start.
    *   *Trade-off:* While `koanf` is initialized, its broader capabilities for hierarchical configuration (e.g., from files, env vars) are not fully leveraged in `initFlags`, which primarily binds to `pflag`. This might be an area for future enhancement for more flexible configuration.
*   **Sequential Service Startup:** Services (Mux, Store, HTTP, Cluster) are initialized in a specific order. This is typical for applications with dependencies (e.g., the Raft transport needs the Mux, the Store needs the Raft transport).
    *   *Trade-off:* A failure in an early critical service (like `net.Listen` or `startNodeMux`) will prevent the application from starting, which is generally desirable.
*   **Custom TCP Multiplexer (`tcp.Mux`):** Allows multiple logical network services (Raft, cluster RPCs) to share a single underlying TCP port. This can simplify firewall configurations and reduce the number of open ports.
    *   *Trade-off:* Adds a layer of complexity compared to using separate ports for each service. The `tcp.Mux` needs to correctly dispatch incoming connections.
*   **Context for Shutdown:** The main `context.Context` derived from signals is passed down to services like `startHTTPService` and `createCluster`, allowing them to react to shutdown requests.
*   **Explicit Raft and Cluster Services:** Separating Raft logic (within `store.NodeStore`) from more general cluster membership and metadata exchange (`cluster.Service`) is a common pattern, allowing for different communication needs.
*   **Deferred Cleanup:** `defer logFile.Close()` ensures the log file is closed on exit. Other cleanup (service closing) is handled explicitly before `main` returns.

# üìñ Technical Documentation: `store` (package `internal/new/store`)

---

### 1Ô∏è‚É£ Overview & Purpose

*   **High-Level Summary:** The `store` package (located at `internal/new/store/`) is responsible for providing a distributed, fault-tolerant key-value store using the Raft consensus algorithm. It manages the node's participation in the Raft cluster, handles data replication, log management, snapshotting, and provides the necessary interfaces for network transport and persistent storage required by the Raft library.
*   **Key Features:**
    *   Raft-based distributed consensus for data storage.
    *   Node lifecycle management within the Raft cluster (bootstrap, join, remove).
    *   Implementation of `raft.FSM` (Finite State Machine) for applying commands.
    *   Implementation of `raft.LogStore` for persistent log storage.
    *   Implementation of `raft.StableStore` for persistent storage of Raft metadata (e.g., current term, voted for).
    *   Snapshotting mechanism (`raft.FSMSnapshot`, `raft.SnapshotStore`) for log compaction.
    *   Network transport abstraction for Raft communication, including gzip compression for snapshots.
    *   Direct interaction with a backend database (e.g., BadgerDB) for the FSM state and Raft logs/stable store.

---

### 2Ô∏è‚É£ Core Concepts & Architecture

*   **High-level architecture:** The `NodeStore` is the central component, embedding and orchestrating a `raft.Raft` instance. It uses a configurable backend `db.DbStore` (with `badgerdb` being a concrete implementation used) for both the Raft log/stable storage and the application's key-value data (FSM state). It exposes methods for cluster operations (join, notify, remove), data manipulation (though specific Set/Get for user data are via `StoreInDatabase`/`GetFromDatabase` which directly access the FSM's DB), and status reporting.
*   **Raft Implementation:**
    *   **FSM (`NodeStore` itself):** The `Apply` method processes Raft logs. In this implementation, it appears to store the raw Raft log entry into its `badgerdb` instance, keyed by the log index.
    *   **LogStore (`NodeStore` embedding `db.DbStore`):** Persists Raft log entries. `raft.LogCache` is used for in-memory caching on top of the persistent store.
    *   **StableStore (`NodeStore` embedding `db.DbStore`):** Persists critical Raft metadata like current term and votedFor.
    *   **SnapshotStore (`raft.FileSnapshotStore`):** Manages snapshots on disk. `FSMSnapshot` and `Snapshot` types wrap the core Raft snapshotting interfaces to add logging, finalizers, and progress monitoring.
    *   **Transport (`NodeTransport` wrapping `raft.NetworkTransport`):** Handles network communication between Raft nodes. It customizes snapshot installation by adding gzip compression/decompression.
*   **Data Storage:**
    *   The FSM's state (application data) is stored in a `badgerdb.DB` instance within `NodeStore`.
    *   Raft logs and stable store data are also persisted using an underlying `db.DbStore` (which can be BadgerDB).
*   **Architectural Pattern:** The package acts as a stateful service layer built around the Raft consensus algorithm. It abstracts the complexities of distributed consensus from higher-level application logic.
*   **Main Components:**
    *   `NodeStore`: The core struct managing Raft, FSM, storage, and cluster operations.
    *   `Config`: Configuration for the `NodeStore` (directory, ID, database type).
    *   `Server`: Represents a Raft server (node) in the cluster.
    *   `Layer`: Interface for the network transport layer consumed by Raft.
    *   `Transport`, `NodeTransport`: Implementations and wrappers for Raft network communication.
    *   `FSMSnapshot`, `Snapshot`: Wrappers for Raft snapshotting.
    *   `db.DbStore`: Interface for the underlying persistent key-value store (e.g., BadgerDB).

**Jules:** Hi! It looks like you're on the right track with using `react-image-gallery`. The issue likely lies in how your image paths are being resolved.

When you use paths like `"path/to/image1.jpg"` in your JSON, the browser tries to fetch these images relative to the current URL of your application. If your application is served from the root (e.g., `http://localhost:3000/`), and your `products.json` is in `public/data/products.json`, then the browser will try to fetch `http://localhost:3000/path/to/image1.jpg`, which is probably not where your images are located.

Here's how you can fix this, assuming your images are in the `public` folder:

1.  **Place your images in the `public` folder:** If they aren't already, move your images into a subfolder within your `public` directory. For example, `public/images/image1.jpg`.

2.  **Update the image paths in your JSON:** Modify your `products.json` to use paths relative to the `public` folder.

    ```json
    [
        {
            "id": 1,
            "title": "Product 1",
            "description": "This is a sample product description for product 1.",
            "price": 100,
            "image": "/images/image1.jpg"
        },
        {
            "id": 2,
            "title": "Product 2",
            "description": "This is a sample product description for product 2.",
            "price": 200,
            "image": "/images/image2.jpg"
        }
    ]
    ```
    Notice the leading `/` in the image paths. This tells the browser to look for the images from the root of your application's public assets.

3.  **Verify your `fetch` path for the JSON:** Ensure that `fetch('/data/products.json')` correctly points to your JSON file. If your `products.json` is in `public/data/products.json`, this path should work correctly as files in the `public` folder are served from the root.

**Here's the corrected `ProductList.js` (no changes needed here if the JSON paths are correct):**

```jsx
import React, { useState, useEffect } from 'react';
import ImageGallery from 'react-image-gallery';
import 'react-image-gallery/styles/css/image-gallery.css';

const ProductList = () => {
  const [products, setProducts] = useState([]);

  useEffect(() => {
    fetch('/data/products.json') // Assuming products.json is in public/data/
      .then(response => {
        if (!response.ok) {
          throw new Error(`HTTP error! status: ${response.status}`);
        }
        return response.json();
      })
      .then(data => setProducts(data))
      .catch(error => console.error('Error fetching products:', error));
  }, []);

  const images = products.map(product => ({
    original: product.image,
    thumbnail: product.image,
    description: product.title,
  }));

  return (
    <div>
      <h1>Product List</h1>
      {images.length > 0 ? (
        <ImageGallery items={images} />
      ) : (
        <p>Loading products...</p>
      )}
    </div>
  );
};

export default ProductList;
```

**Key things to check:**

*   **Image Paths:** Double-check that the paths in your `products.json` now correctly point to the images within your `public` folder (e.g., `/images/image1.jpg`).
*   **`public` Folder:** Ensure your images are indeed inside the `public` folder (or a subfolder within it). Create React App (and similar setups) serve files from this directory directly.
*   **Network Tab:** Open your browser's developer tools (usually by pressing F12) and go to the "Network" tab. Refresh the page. Look for any 404 (Not Found) errors related to the image URLs. This will tell you exactly what path the browser is trying to load and failing.
